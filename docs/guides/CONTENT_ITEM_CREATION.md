# Content Item 作成ガイド

**作成日**: 2026-02-03

---

## 1. ページタイプの定義

Content Itemは以下の3種類に分類される。**1ページ = 1目的**を厳守する。

| タイプ | 目的 | 一言で言うと |
|--------|------|-------------|
| **Service Page** | 行動させる | やらせる |
| **Guide Page** | 理解させる | 理解させる |
| **Answer Page** | 判定する | 答えを出す |

---

## 2. Service Page（行動ページ）

### 定義

**利用者に具体的な行動を起こさせることを唯一の目的とするページ。**

行動とは、申請・登録・支払い・予約・提出・外部フォーム遷移などを指す。

### 目的

- 「今すぐ何をすればよいか」を明確に示す
- 読ませず、迷わせず、押させる

### Serviceの原則（絶対ルール）

```
文章ほぼなし
スクロールさせない
CTAが主役
```

---

### 2.1 ページ構成（上から下）

#### 1. タイトル（動詞で始まる）

- 例：「住民票の写しを請求する」「転入届を届け出る」「児童手当を申請する」

**ルール**:
- 必ず動詞で始まる
- 「〜について」「〜の案内」は禁止

#### 2. 要約（1〜2行）

- 誰が・何をするかだけ
- 例：「市内に住民登録のある方が、住民票の写しを窓口で請求する手続きです。」

**文字数目安**: 40〜60字

#### 3. 最小限の前提条件

- 対象者（箇条書き）
- 必要なもの（箇条書き）
- 手数料（1行）

**ルール**:
- 各項目3つ以内
- 詳細はGuideへ

#### 4. メインCTA（ActionButton）

- 「申請を開始する」「オンラインで請求する」「予約する」
- ページの中央〜下部に配置
- 最も目立つ要素

#### 5. 補足リンク

- 「詳しく知りたい方はこちら」→ Guide
- 「対象か確認する」→ Answer

---

### 2.2 Serviceが絶対にやらないこと

| 禁止事項 | 理由 |
|---------|------|
| 制度の背景説明 | → Guideの役割 |
| 長文の注意書き | 行動の妨げ |
| 判定質問（Yes/No） | → Answerの役割 |
| FAQ・よくある質問 | 行動の妨げ |
| StepNavigation（手順の詳細） | 長すぎる |

---

### 2.3 対応コンポーネント

```
Breadcrumbs
Title（動詞で始まる）
Summary（1-2文、40-60字）
NotificationBanner?（重要な注意1つだけ）
Section: 対象者（箇条書き、3項目以内）
Section: 必要なもの（箇条書き、3項目以内）
Section: 手数料（1行）
ActionButton（CTA）
Contact
RelatedLinks（Guide / Answer へのリンク）
```

**使わないコンポーネント**:
- Table（→ シンプルなリストで十分）
- StepNavigation（→ 長すぎる）
- Accordion（→ 行動の妨げ）

---

### 2.4 Serviceページ作成チェックリスト

- [ ] タイトルは動詞で始まるか
- [ ] サマリーは1-2行（40-60字）で完結しているか
- [ ] 前提条件は各3項目以内か
- [ ] ActionButtonがあるか
- [ ] CTAが最も目立つ要素か
- [ ] 長文の説明を書いていないか
- [ ] Guide / Answer へのリンクがあるか

## 3. Guide Page（理解ページ）

### 定義

**判断や行動の「前提理解」をつくるページ。**

決めない・始めさせない。理解させて、次に送るのが仕事。

### 目的

- 制度の全体像を把握させる（熟読させない）
- 見出しの流し読みで理解できる構造にする
- Answer / Service へ送り出す

### Guideの原則（絶対ルール）

```
読ませない前提で書く
見出しが8割
文章は補足
判断はさせない
```

---

### 3.1 ページ構成（上から下）

#### 1. タイトル（名詞・制度名）

- 例：「児童手当」「国民健康保険」「住民票」
- **禁止**: 疑問文・動詞で始めない
- **目的**: 「これは何の話か」を即認識させる

#### 2. 冒頭サマリー（2〜3行）

その制度・仕組みの要点だけを書く:
- 誰向けか
- 何が起きるか

**ルール**:
- ここだけ読めば概要が分かる
- 条件・例外は書かない
- **文字数目安**: 60〜100字

#### 3. セクション分割（見出しが主役）

Guideの本体。**見出しだけ読んでも全体像が掴める**ように設計する。

**見出しの例**:
| 良い見出し | 悪い見出し |
|-----------|-----------|
| 〜とは | 概要 |
| 受け取れる人 | 対象者について |
| いくら受け取れるか | 支給額 |
| いつ届くか | 支給日 |
| 申請が必要なとき | 申請について |

**ルール**:
- 見出し＝質問への短い答え
- 抽象見出し（概要・詳細・その他）は**禁止**
- 見出しを並べただけでFAQになる構造

---

### 3.2 各セクションの中身

#### 構造

```
見出し
短い説明（1〜2文）
箇条書き（条件・要点）
```

#### 文字量の制限

| 要素 | 文字数 |
|------|--------|
| 1セクション全体 | 80〜120字 |
| 説明文 | 1〜2文（40字以内） |
| 箇条書き1項目 | 20字以内 |

**ルール**:
- 4行を超えたら分割
- 「スクロールしても、読む量は増えない」設計

---

### 3.3 Guideが絶対にやらないこと

| 禁止事項 | 理由 |
|---------|------|
| 対象かどうかの最終判断 | → Answerの役割 |
| Yes / No の断定 | → Answerの役割 |
| 申請開始ボタン（ActionButton） | → Serviceの役割 |
| 手続きの詳細ステップ（StepNavigation） | → Serviceの役割 |
| FAQ・よくある質問（Accordion） | 長文化の原因 |
| 長文の背景説明 | 読まれない |

---

### 3.4 他ページとの接続（出口）

**Guideは必ず分岐点を持つ。単体で完結させない。**

| 出口の種類 | リンク先 | 例 |
|-----------|---------|-----|
| 「対象か確認する」 | Answer | 「受け取れるか確認する」 |
| 「申請する」 | Service | 「申請手続きを開始する」 |
| 「詳細な例外」 | 別Guide | 「公務員の場合」 |

**配置**: RelatedLinks ブロックで、ページ下部に配置

---

### 3.5 ユーザーの操作体験

```
1. 検索でGuideに来る
2. 冒頭サマリーで「読む価値」を判断
3. 見出しを流し読み
4. 自分に関係ある所だけ拾う
5. 判定 or 行動へ移動
```

**熟読は想定していない。**

---

### 3.6 対応コンポーネント

```
Breadcrumbs
Title（名詞）
Summary（2-3行、60-100字）
Section: 〜とは（1-2文）
Section: 受け取れる人（箇条書き）
Section: いくら受け取れるか（箇条書き）
Section: いつ届くか（1文）
Section: 申請が必要なとき（箇条書き）
NotificationBanner?（重要な注意1つだけ）
RelatedLinks（出口：Answer / Service へのリンク）
Contact
```

**使わないコンポーネント**:
- ActionButton（→ Serviceで使う）
- StepNavigation（→ Serviceで使う）
- Accordion（→ 長文化の原因）
- Table（→ 比較が必要な場合のみ）
- Sources（→ シンプル化のため省略可）

---

### 3.7 Guideページ作成チェックリスト

- [ ] タイトルは名詞（制度名）か
- [ ] サマリーは2-3行（60-100字）で完結しているか
- [ ] 見出しだけ読んで全体像が分かるか
- [ ] 各セクションは80-120字以内か
- [ ] 4行を超えるセクションがないか
- [ ] ActionButton を使っていないか
- [ ] StepNavigation を使っていないか
- [ ] Accordion を使っていないか
- [ ] 出口（RelatedLinks）があるか
- [ ] 出口に Answer / Service へのリンクがあるか

## 4. Answer Page（判定ページ）

### 定義

**読むページではない。操作して「判定結果を出す」ページ。**

文章理解を要求せず、入力 → 分岐 → 結果だけを提供する。

### 目的

- 考えさせずに結論を出す
- 対象外の人を早期に振り分ける

### Answerの原則（絶対ルール）

```
説明しない
教えない
読ませない
迷わせない
"決める"ことだけに集中
```

---

### 4.1 ページ構成（上から下）

#### 1. 質問タイトル（h1）

- 質問文そのものがタイトル
- 例：「児童手当を受け取れますか？」「住民票が必要ですか？」

**ルール**:
- 名詞NG、必ず疑問文
- 1つの判断軸だけ

#### 2. 現在の質問（1問だけ表示）

- Yes / No ボタン
- または選択肢ボタン

**ルール**:
- 同時に2問出さない
- 説明文は最小限（あっても1行）
- **肯定的な選択肢（「はい」等）は必ず左側（配列の先頭）に配置する**
- 質問文は「はい」が対象者に有利な回答になるように設計する
  - 良い例: 「18歳の年度末までですか？」→「はい」で次へ進む
  - 悪い例: 「18歳の年度末を過ぎていますか？」→「いいえ」で次へ進む

#### 3. ナビゲーション操作

- 「前の質問に戻る」ボタン（必須）
- 進行状況は見せない or 最小

---

### 4.2 内部ロジック（条件分岐ツリー）

```
Q1: 年齢は？
 ├─ 18歳未満 → Result A
 └─ 18歳以上 → Q2

Q2: 住民登録は？
 ├─ 市内 → Q3
 └─ 市外 → Result B

Q3: 公務員ですか？
 ├─ いいえ → Result C（対象）
 └─ はい → Result D（勤務先で申請）
```

**ルール**:
- 完全に決定論（曖昧な文言は禁止）
- 例外はGuideへ逃がす

---

### 4.3 結果ページの構成（最重要）

#### Result（結論）は必ず最上部

- 「受け取れます」
- 「受け取れません」
- 「勤務先で申請してください」

**結論が1行で分かる。**

#### 次の行動（必須）

| 結果 | リンク先 |
|------|---------|
| 対象 | Service（「申請手続きを開始する」） |
| 理由を知りたい | Guide（「詳しく見る」） |
| 対象外 | 関連制度へのリンク |

**Answerは必ず次に送る。**

#### 補足（必要な場合のみ）

- なぜそうなったか（1〜2行）
- 注意点（箇条書き）

**ここで長文を書いたら失敗。**

---

### 4.4 ユーザーの操作体験

```
1. 検索から直行
2. 1問目に答える
3. 数問で分岐
4. 即、結論が出る
5. 行動 or 離脱（どちらも成功）
```

**「結果を見て離脱」も成功体験。**

---

### 4.5 Answerが絶対にやらないこと

| 禁止事項 | 理由 |
|---------|------|
| 制度の詳細解説 | → Guideの役割 |
| 長文の背景説明 | 読まれない |
| 網羅的な注意事項 | 判定が遅れる |
| 複数の質問を同時表示 | 迷う |
| 進行状況バー | 不要 |

---

### 4.6 対応コンポーネント

```
Breadcrumbs
Title（質問形式：〜ですか？）
SmartAnswer（質問フロー + 結果表示）
```

**SmartAnswerブロックの構造**:
```json
{
  "type": "SmartAnswer",
  "props": {
    "questions": [
      {
        "id": "q1",
        "text": "質問文",
        "options": [
          { "label": "はい", "nextQuestionId": "q2" },
          { "label": "いいえ", "resultId": "result-no" }
        ]
      }
    ],
    "results": [
      {
        "id": "result-yes",
        "title": "受け取れます",
        "description": "1-2行の説明",
        "actionLabel": "申請手続きを開始する",
        "actionHref": "/service-page",
        "relatedLinks": [
          { "title": "詳しく見る", "href": "/guide-page" }
        ]
      }
    ]
  }
}
```

---

### 4.7 Answerページ作成チェックリスト

- [ ] タイトルは疑問文か
- [ ] 1問ずつ表示されるか
- [ ] 「前の質問に戻る」があるか
- [ ] 結果は1行で結論が分かるか
- [ ] 結果に次の行動リンクがあるか
- [ ] 長文の説明を書いていないか
- [ ] SmartAnswerブロックを使っているか

## 5. ページタイプ判定フロー

以下の質問に**上から順に**答え、最初に確定したものを採用する。

### STEP 1：結論の出し方

```
Q1. この情報は Yes / No / 数値 で結論を出せるか？
    → YES: Q2へ
    → NO:  Guide Page

Q2. 結論は条件（年齢・収入・居住地など）で変わるか？
    → YES: Answer Page
    → NO:  Q3へ

Q3. 結論を誤解すると利用者に不利益が生じるか？
    → YES: Guide Page
    → NO:  Answer Page
```

### STEP 2：行動がゴールか

```
Q4. このページのゴールは申請・登録・支払い・予約などの行動か？
    → YES: Service Page
    → NO:  Q5へ

Q5. 主要要素はCTAボタンか？
    → YES: Service Page
    → NO:  Q6へ
```

### STEP 3：理解が必要か

```
Q6. 制度や仕組みを理解しないと判断できないか？
    → YES: Guide Page
    → NO:  Q7へ

Q7. 複数の選択肢・例外・注意点が存在するか？
    → YES: Guide Page
    → NO:  Answer Page
```

### 判定フロー図

```
                    ┌─────────────────┐
                    │  コンテンツ分析  │
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
         ┌────▼────┐    ┌────▼────┐    ┌────▼────┐
         │ 行動が  │    │ 判定が  │    │ 理解が  │
         │ ゴール？│    │ ゴール？│    │ ゴール？│
         └────┬────┘    └────┬────┘    └────┬────┘
              │              │              │
         ┌────▼────┐    ┌────▼────┐    ┌────▼────┐
         │ Service │    │ Answer  │    │  Guide  │
         │  Page   │    │  Page   │    │  Page   │
         └─────────┘    └─────────┘    └─────────┘
```

## 6. 禁止ルール

### 絶対禁止

| 禁止事項 | 理由 |
|---------|------|
| 1ページに複数の目的を持たせる | ユーザーが迷う |
| Service Page で説明する | 行動の妨げになる |
| Answer Page で解説する | 判定が遅れる |
| Guide Page で判定させる | 理解の前に判断させてしまう |

### ページタイプ別の禁止事項

| タイプ | 禁止 |
|--------|------|
| Service | 長文説明、FAQ、判定質問 |
| Guide | 強制的CTA、Yes/No判定UI |
| Answer | 詳細解説、網羅的注意事項 |

## 7. content_type との対応

INNOMAスキーマの `content_type`（3種類のみ）:

| ページタイプ | content_type | タイトル形式 |
|-------------|--------------|-------------|
| Service Page | `service` | 動詞で始まる（〜を申請する） |
| Guide Page | `guide` | 名詞（制度名・テーマ名） |
| Answer Page | `answer` | 質問形式（〜ですか？） |

### service_category

タクソノミーから柔軟に設定（string型）。例:
- `registration` - 届出・証明
- `childcare` - 子育て・出産
- `health` - 健康・医療
- `tax` - 税金
- その他、タクソノミーで定義されたカテゴリ

## 8. 作成チェックリスト

Content Item 作成時は以下を確認する:

### 8.1 ページタイプの確定

- [ ] 判定フロー（セクション5）に従ってタイプを確定した
- [ ] 1ページ = 1目的 を満たしている
- [ ] 禁止ルール（セクション6）に違反していない

### 8.2 コンテンツ整合性

- [ ] タイトルと内容が一致している
- [ ] タイトルがページタイプに適した形式になっている
  - Service: 動詞で始まる
  - Guide: 名詞（制度名）
  - Answer: 質問形式
- [ ] Summary がページ内容を正しく要約している

### 8.3 変数・RichText の妥当性

- [ ] 静的コンテンツが全国共通で正確か
- [ ] 自治体ごとに異なる情報が変数化されているか
- [ ] `template-validity-checker` エージェントで検証済み

### 8.4 コンポーネント最適化

- [ ] ページタイプに適したコンポーネント構成か
- [ ] [COMPONENT_SELECTION_LOGIC.md](./COMPONENT_SELECTION_LOGIC.md) のルールに従っているか
